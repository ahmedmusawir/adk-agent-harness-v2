### [Morning] — ADK ContextCacheConfig Research
- Tony received Google info on `ContextCacheConfig` — token-level prompt caching for system prompt + tool schemas
- Clarified two concepts Tony was mixing:
  1. `adk web` session "caching" = `InMemorySessionService` turn history in RAM — NOT token savings
  2. `ContextCacheConfig` = model-side token cache for static prefix (system prompt + tool schemas) — 90% cost reduction on cached tokens, up to 85% latency reduction
- Decision: **hold off on ContextCacheConfig** until after manual smoke test and custom UI work
- Custom UI note: will require explicit session service setup (`InMemorySessionService` for dev, `VertexAiSessionService` for prod) + manual session ID management. GCS session memory tools (agent-level) are separate from turn-history session management — both needed.

## Lessons Learned
- `adk web` InMemorySessionService = conversation history, NOT prompt caching — two different things
- ContextCacheConfig wraps agent in an `App` object — different deployment model from `adk web .`, verify import paths before using
- Usage report "no data" is expected on first run of a new day — not a bug
