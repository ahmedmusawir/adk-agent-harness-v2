## Lessons Learned
- ADK web UI Eval tab is hardcoded to 2 metrics in compiled Angular JS (`var acA = [{metricName: "tool_trajectory_avg_score"...}]`). The `MetricEvaluatorRegistry` and `/metrics-info` endpoint are newer additions that the frontend never adopted. Custom metrics work via the Python API but not the UI.
- `_registry` in `MetricEvaluatorRegistry` is a class-level mutable dict ‚Äî all instances share it. Registration on `DEFAULT_METRIC_EVALUATOR_REGISTRY` is globally visible.
- The `Evaluator` base class has sync `evaluate_invocations`, but `LlmAsJudge` overrides it as async. Runner must check `inspect.iscoroutinefunction()` to handle both (same pattern as `LocalEvalService`).
- Deterministic tool-presence checks are the right first step ‚Äî they're free (no API cost), fast, and prove the infrastructure works before spending tokens on LLM judges.
- The runner script must load the agent's `.env` file ‚Äî `adk web` does this automatically but standalone scripts don't. Fixed by adding `dotenv.load_dotenv()`.
- LLM judge prompts need careful scoping ‚Äî the `stays_in_role` judge flagged "What time is it?" as off-topic trivia, even though answering time questions is a designed capability. Judge prompt needs a carve-out for utility questions.
- Every metric runs against every case ‚Äî not 1:1. A metric like `engineer_prompt_format` will naturally score low on cases that aren't about writing prompts. This is expected, not a bug.
