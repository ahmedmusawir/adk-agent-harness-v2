## Lessons Learned
- **Always read the installed SDK before writing against it.** The `.name` attribute bug on `AgentEngine` was a guess that held up in my head but not in the code. Five minutes grepping `vertexai/_genai/types.py` would have prevented the failed first run.
- **GCP resource names use project NUMBER, not project ID.** Don't be surprised by the numeric prefix — same project.
- **`AgentEngine` wrapper ≠ `ReasoningEngine` resource.** Always go through `.api_resource` for resource metadata.
- **LRO wrapping:** `operation.response` contains the hydrated resource after `wait_for_completion=True` — but only for `memories.create()`. `memories.generate()`'s response is SKELETAL (name-only); caller must `get()` afterward.
- **`ExperimentalWarning` on `vertexai.agent_engines.*` is benign.** Expect it on every call; don't flag as a bug.
- **Memory Bank's generation LLM is NOT a per-call parameter.** It's wired at engine provision via `context_spec.memory_bank_config.generation_config.model`. If we want control, that's set once, not per-request.
- **Consolidation defaults to ON** (`disable_consolidation=None`). Useful for production but muddies signal during discovery — choose test content carefully to separate extraction-behavior findings from consolidation-behavior findings.
- **Memory Bank extraction is SELECTIVE, not exhaustive.** Do not assume every user statement becomes a memory — the default model silently drops facts it doesn't consider salient. In our test, 1 of 2 facts was dropped (the named-entity location fact).
- **Generated/consolidated facts are always FIRST-PERSON.** Voice normalization is one-way: regardless of whether the prior text was third-person, any write via `generate()` (including UPDATED mutations) comes out first-person. Manual `create()` writes keep whatever voice the caller passes.
- **Consolidation is IN-PLACE and PRESERVES HISTORY.** When a new fact contradicts an existing memory, the service issues `UPDATED` on the same memory ID and rewrites the fact to capture both the current state AND the historical context (e.g., "I now use X; previously, I preferred Y."). No delete-then-create; memory IDs are stable.
- **No client-side dedup needed for Jarvis integration.** Trust server-side consolidation to maintain coherent memory state. Just feed it transcripts.
